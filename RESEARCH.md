## 产品定位

一个面向 AI 从业者和爱好者的信息聚合平台，每日自动汇集 GitHub 热榜、arXiv 最新论文、AI 资讯和深度播客。不只是聚合链接，更通过**标签体系**和**热门议题聚合**帮助用户构建个人认知体系。

**一句话价值：** 每日自动汇集 GitHub 热榜、AI 论文、行业资讯与深度播客，通过智能标签与热门议题聚合，让 AI 从业者高效掌握行业动态。

**目标用户：** 追求信息效率和认知深度的 AI 专业人士

### 数据库说明：PostgreSQL（Supabase）

- 使用 **Supabase 免费版**（500MB 存储），内置 PostgreSQL，SSL 连接
- 连接方式：`DATABASE_URL` 环境变量（连接池字符串）
- 使用 `pg`（node-postgres）驱动，连接池模式
- **关键约定：RSS 抓取任务顺序执行（非并行）**，避免数据库连接争抢
- MVP 数据量预估：每天 4 次抓取，每次 ~50 条，月均 ~6000 条，500MB 绰绰有余

### 播客 RSS 特殊说明

Substack/Fireside 的播客 RSS 与普通 RSS 格式有差异，解析时需注意：

| 字段 | 普通 RSS | 播客 RSS |
|------|---------|----------|
| 内容容器 | `<item>` | `<item>` + `<enclosure>` |
| 摘要 | `<description>` | `<description>`（可能含 HTML） |
| 音频链接 | 无 | `<enclosure url="..." type="audio/mpeg"/>` |
| 封面 | 无 | `<itunes:image href="..."/>` |

**解析方式：** 检测 `<enclosure type="audio/*">` 时按播客逻辑分支处理。

**异常处理：**
- **多 enclosure：** 取首个 `type="audio/mpeg"` 或 `type` 包含 `audio` 的链接
- **无有效 enclosure：** 卡片退化为普通文章样式，隐藏播放按钮，不影响页面渲染
- **播客摘要含 HTML：** 提取纯文本内容用于展示

### RSSHub 备选方案

| 优先级 | 方案 | 适用数据源 |
|--------|------|-----------|
| 主方案 | RSSHub 公共实例 `https://rsshub.app` | GitHub Trending 等 |
| 备用 1 | RSSHub 备用实例（如 `rsshub.bili.xyz`） | 同上 |
| 备用 2 | 官方源直连 | arXiv / 机器之心 / Substack 播客 |
| 备用 3 | GitHub API（保底方案） | GitHub Trending |

**代码实现：** 每个数据源配置 `primary_url` + `fallback_urls[]`，主 URL 失败时自动顺次尝试备用。

### 更新策略

每日 4 次通过 node-cron 触发抓取：**8:00 / 12:00 / 18:00 / 22:00（北京时间，UTC+8）**

每次抓取逻辑：
1. 顺序遍历启用的数据源（非并行），逐个请求 RSS feed
2. 解析 XML，以 URL 去重
3. 通过关键词词典自动匹配并打标签（匹配范围：标题 + 摘要 + `content:encoded` 前 200 字）
4. 新内容写入 PostgreSQL，已有内容跳过
5. 单源失败记录日志并尝试备用地址，不影响其他源

### 标签体系

**打标方式：** 关键词自动匹配（规则引擎，不调 AI）

**匹配范围：** 标题 + 摘要 + `content:encoded`（取前 200 字）。覆盖大部分 RSS 源的实际内容。

**匹配规则：**
- 中文关键词（如"大模型"）天然无边界问题，直接匹配
- 英文关键词统一使用正则 `new RegExp('\\b' + keyword + '\\b', 'i')`，确保词边界且大小写不敏感
- 避免 `Agent` 误配 `Agents`、`LLM` 误配 `LLaMA` 或无关单词

**词典存储：** 存放在数据库 `tag_keywords` 表中，不写在代码里。提供 `POST /api/admin/retag` 接口，修改词库后触发全量重新打标。

**步骤：**
1. 共建 AI 领域关键词词典（确认初始词库后预填入数据库）
2. 每次抓取 RSS 时，对标题 + 摘要 + content:encoded 前 200 字做关键词匹配
3. 一条内容可打多个标签
4. 后续修改词库后调用 retag 接口全量重打（几千条数据 < 1s）

### 热度评分算法（伪热度，MVP 简化版）

卡片上展示的"热度 90°C"为视觉元素，后端定义伪热度公式：

```
hot_score = base_score × recency_boost

base_score（数据源基础权重）:
  - GitHub Trending: 65（基础分）+ star_count / 10000（增量）
  - arXiv 论文: 55（基础分，无引用数据时）
  - 机器之心: 60（中文 AI 媒体权重）
  - 新智元: 50
  - Lenny's Podcast: 70（独特性加成）
  - 管理员爆料: 75（人工录入默认较高）

recency_boost（时间衰减）:
  - 12 小时内: 1.5
  - 24 小时内: 1.3
  - 48 小时内: 1.1
  - 超过 48 小时: 1.0（不再额外加分）
```

**前端展示：** `hot_score` 映射到 0~100°C 范围显示。映射方式：**线性等比映射**，当前批次最高分 = 100°C，其他按比例折算。MVP 版本热度主要用于排序，温度数值是排序的直观表达，不追求精确量化。

### 热门议题聚合

**方案：基于关键词共现的轻量聚合（无需 NLP 模型）**

```
抓取入库 → 提取标题/摘要中的已定义关键词 → 统计跨源共现 → 热度排序 → 聚合展示
```

**具体逻辑：**
1. 每次新文章入库时，提取标题/摘要中匹配的关键词（复用标签词库）
2. 统计**最近 48 小时（或最近 2 次抓取批次）**内，同一关键词在 ≥2 种不同数据源类型中出现的情况
3. 聚合阈值：**同一关键词在 ≥2 个不同源类型中出现，且总文章数 ≥3 条**
4. 聚合展示形式：议题卡片采用 **双行标题** 机制：
   - **主标题：** 从聚合的文章中选取热度最高或标题最吸引人的一篇，作为议题主标题
   - **副标题：** 标注 `话题：#Agent · 共 X 篇跨源探讨`，并展示各数据源的分布
5. 避免"标题垃圾化"：不直接把关键词当作议题标题

**示例：**
```
┌─────────────────────────────────────────────┐
│  AgentKit：一个轻量级AI Agent框架           │  ← 主标题（取热度最高文章）
│  话题：#Agent · 共 4 篇跨源探讨             │  ← 副标题
│                                             │
│  📦 GitHub   AgentKit 框架                  │
│  📄 arXiv    Agent 推理优化论文              │
│  📰 机器之心  Agent 落地分析文章              │
│  📰 新智元    Agent 应用案例                 │
└─────────────────────────────────────────────┘
```

**阈值调整：** MVP 先用 `≥2 个源且 ≥3 条`，实际数据出来后再根据效果微调。

### 管理员爆料

- 简易隐藏页面（`/admin`），环境变量 `ADMIN_TOKEN` 鉴权
- **鉴权流程：**
  1. 访问 `/admin` 页面时，先显示一个简易 Token 输入框（类似登录页）
  2. 管理员手动输入 `ADMIN_TOKEN`，前端存入 `sessionStorage`（关闭页面即清除）
  3. 后续每次请求 `/api/admin/*` 时，从 `sessionStorage` 读取 Token，放在 `Authorization: Bearer <TOKEN>` Header 中
  4. 后端解析 Header，对比环境变量中的 `ADMIN_TOKEN`，一致则放行，否则返回 403
  5. 失败 3 次后前端显示"请刷新页面重试"（简单防爆破）
- **Token 从不存储在前端代码中**，只在管理员手动输入时存在于 sessionStorage（页面级会话）
- 无防爆破机制（MVP 阶段可接受，后续加入 Rate Limit）
- 手动填写标题/链接/摘要/分类/配图
- **提交后默认进入"今日热点"feed**，卡片上带特殊 `[爆料]` 红色角标，与其他内容区分
- 无审核流，无用户系统
- 修改标签词库后，可在管理页面触发 `POST /api/admin/retag` 全量重新打标

## 风险与约束

1. **RSS 源变更**：上游 RSS 格式变更或限流 → 失败时记录日志，尝试备用地址，不影响其他源
2. **RSSHub 公共实例不稳定**：配置 `primary_url` + `fallback_urls[]`，自动故障转移；后续可自建 RSSHub
3. **arXiv RSS 偶有延迟**：缓存 6 小时，不影响展示
4. **版权合规**：仅展示标题与摘要（< 200 字），点击新窗口跳转原文；标明数据来源，非商用
5. **PostgreSQL 并发**：PostgreSQL 原生支持并发读写，无需特殊配置。连接池通过 `pg` 驱动管理
6. **Render 休眠策略**：服务 15 分钟无流量休眠（冷启动约 30s），数据库外置在 Supabase 不受影响
7. **播客 RSS 格式差异**：需单独解析 `<enclosure>` 和 `<itunes:*>` 字段
8. **英文关键词词边界**：`LLM` 可能误配 `LLaMA`。MVP 直接使用正则 `new RegExp('\\b' + kw + '\\b', 'i')` 匹配，避免误配
